## 1. HTTP와 HTTPS의 차이점은 무엇인가?

- HTTP와 HTTPS는 인터넷에서 데이터를 주고 받는데 사용하는 프로토콜입니다.
- 둘의 가장 큰 차이점은 보안성, 무결성 입니다.
- HTTP는 데이터를 암호화하지 않고 전송합니다. 따라서 중간에 데이터가 탈취 됐을 시, 악용될 여지가 있습니다. <br />
또한 데이터가 전송 중에 변조, 손상되지 않았음을 보장하지 않습니다.(무결성 보장 X)
- HTTPS는 SSL, TLS 프로토콜을 사용하여 데이터를 암호화합니다. 이를 통해 데이터의 보안성을 높이고 무결성도 보장할 수 있습니다. <br />
또한 SSL, TLS 인증서를 통해 웹 사이트의 신원을 인증할 수 있습니다. 이를 통해 사용자는 웹 사이트가 신뢰할만 한 곳인지 확인할 수 있습니다. <br />

<br />

### 1-1. 프로토콜이란 무엇인가?

- 프로토콜은 서로 다른 시스템 간에 데이터를 주고받는 데 사용되는 약속된 규칙의 집합입니다.
- 데이터가 각 단계에서 어떤 정보가 전송되어야 하는지, 어떤 형식으로, 어떠한 암호화 방식을 사용해야 하는 지 등을 정의합니다.

<br />

### 1-2. SSL과 TLS 프로토콜은 무엇인가?

- SSL과 TLS는 데이터를 보호하기 위한 암호화 프로토콜입니다.
- TLS는 SSL를 대체하기 위해 개발된 프로토콜 입니다.
- 서버는 인증서를 클라이언트에 제공하고, 클라이언트는 신뢰도 확인 및 인증서에 제공된 공개키를 통해 데이터를 암호화하여 전송합니다. <br />
서버는 클라이언트에서 전송된 데이터를 비공개키를 통해 데이터를 복호화합니다. 이러한 방식을 공개키 암호화 방식 혹은 비대칭키 암호화 방식이라 말합니다.

## 2. HTTP 요청 메소드는 무엇인가?

- HTTP 요청 메서드는 클라이언트가 서버로 요청을 보낼 때, 요청의 목적과 방식을 지정하는 데에 사용되는 메서드입니다.
- GET: 서버의 데이터를 조회할 때 사용하는 메서드 입니다.
- POST: 서버에 데이터를 추가하고자 할 때 사용 합니다.
- PUT : 서버의 데이터를 업데이트 하고자 할 때 사용합니다. (전체 데이터를 업데이트 합니다.)
- PATCH: 서버의 데이터 중 일부분만 업데이트 하고자 할 때 사용합니다.
- DELETE: 서버의 데이터를 삭제하고자 할 때 사용합니다.

<br />

### 2-1. 쿼리(query)와 파라미터(params)의 차이는 무엇인가?

- 쿼리와 파라미터는 공통적으로 서버에 특정 조건을 전달하는 매개 변수입니다.
- 쿼리 : 주로 GET 요청에서 사용됩니다. "key=value" 형식으로 URL의 끝 부분에 "?"와 함께 추가됩니다. <br />
데이터를 검색할 때 사용 됩니다. <br />
ex) https://www.example.com/search?q=test
- 파라미터 : "/"를 사용해 URL의 끝 부분에 추가 됩니다. <br />
서버에서 특정 조건을 요청할 때 사용 됩니다. 주로 RESTful API 에서 사용됩니다. <br />
예를 들어, 파라미터를 포함하면 특정 데이터를 삭제 한다는 의미이고, 파라미터가 없다면 특정 데이터를 생성 하는 방식으로 구성할 수 있습니다. <br />
ex) https://www.example.com/items/1234

<br />

### 2-2. URL은 무엇인가? 

- URL은 인터넷 상에서 특정한 리소스의 위치를 가리키는 문자열 입니다.
- URL은 다음과 같이 구성됩니다.
```
scheme://host:port/path/?=query=value#fragment
```
- 스키마(scheme): 프로토콜을 의미합니다.
- 호스트(host): 리소스가 위치한 서버의 도메인 이름 또는 IP 주소를 나타냅니다.
- 포트(port): 호스트와 통신할 때 사용되는 포트 번호를 의미합니다. HTTP의 기본 포트 번호는 80, HTTPS의 기본 번호는 443 입니다. 생략 가능합니다.
- 경로(path): 리소스의 경로를 나타냅니다.
- 쿼리 문자열: 위 설명과 같습니다.
- 프래그먼트: 문서 내의 특정 위치를 나타냅니다.

<br />

### 2-3. 도메인이란 무엇인가?

- 인터넷 상에서 컴퓨터들을 식별하기 위한 이름으로 구성된 계층 구조적인 이름체계 입니다. 
- 각 컴퓨터는 고유한 IP 주소를 가지고 있지만 이를 사람이 기억하기 어렵고, 변경 될 수 있기 때문에, 도메인 이름을 사용합니다.

<br />

### 2-4. IP란 무엇인가?

- 인터넷 프로토콜의 약자로, 인터넷 상에서 컴퓨터들이 서로 정보를 주고 받을 수 있게 하는 프로토콜입니다. 일종의 컴퓨터의 주소라고 볼 수 있습니다. <br />
- IPv4 (32bit 주소 체계), IPv6(128bit 주소 체계)가 있습니다.

## 3. HTTP 상태 코드는 무엇인가?

- HTTP 상태 코드는 HTTP 요청에 대한 응답 상태를 나타내는 3자리 숫자로 구성된 코드 입니다.
- HTTP 상태 코드에 따라 요청이 성공적으로 실행 되었는지, 오류가 발생했는 지를 확인할 수 있습니다.
- 1xx(Informational): 요청이 수신되었으며 처리 중입니다.
- 2xx(Successfull): 요청이 성공적으로 처리되었습니다.
- 3xx(Redirection): 요청을 완료하기 위해 추가 작업이 필요합니다.
- 4xx(Client Error): 클라이언트 측의 오류로 요청이 실패했습니다.
- 5xx(Server Error): 서버 측의 오류로 요청이 실패했습니다.
- 400 Bad Requet: 클라이언트에서 잘못된 요청을 보냈을 때 발생합니다.
- 401 Unauthorized: 인증이 필요할 때 발생합니다.
- 403 Forbidden: 접근이 금지되었을 경우 발생합니다.
- 404 Not Found: 클라이언트에서 잘못된 URL을 요청했을 경우 발생합니다.

## 4. CORS는 무엇인가?

- CORS는 웹 애플리케이션이 다른 출처의 자원에 접근할 수 있는 권한을 부여하는 메커니즘입니다.
- 도메인이 다를 경우 발생합니다.
- 다른 도메인에서 리소스 요청을 자유롭게 할 수 있다면 웹사이트에서 사용되는 쿠키나 세션 등이 탈취될 우려가 있습니다.

<br />

### 4-1. CORS 문제는 어떻게 해결하는가?

- 서버 측에서 "Access-Control-Allow-Origin" 헤더를 설정하여 특정 출처에서 온 요청을 허용할 수 있습니다.

## 5. DNS란 무엇인가?

- DNS란 인터넷에서 도메인 이름을 IP 주소로 변환하는 시스템입니다.
- URL이 주소창에 입력되면, 전세계 서버들 중에서 도메인 이름에 대한 IP 주소 정보를 가지고 있는 서버를 찾아 IP 주소 정보를 가져옵니다.

## 6. TCP/IP 프로토콜이란 무엇인가?

- IP의 설명은 위를 참고하시면 됩니다.
- TCP는 IP 프로토콜을 기반으로 동작하며, 데이터의 전송을 안정적으로 보장하는 프로토콜 입니다.
- 데이터를 패킷 단위로 나누어 전송하고, 각 패킷이 제대로 도착했는지 확인하고, 제대로 도착하지 않은 패킷을 재전송 하는 등의 기능을 제공합니다. (데이터 무결성 보장)
- 또한, 순서 보장의 기능을 제공합니다. TCP는 각 패킷에 순서 번호를 부여하고, 수신자는 이 번호를 통해 패킷을 재조합합니다. 이를 통해 <br />
데이터가 중간에 손실되거나 중복 전송이 되더라도 데이터를 제대로 구성할 수 있습니다.

<br />

### 6-1. 패킷이란 무엇인가?

- 패킷은 데이터를 전송하기 위한 최소 단위 입니다.
- 데이터를 전송할 때에는 데이터를 작은 단위로 나누어 패킷으로 만들고, 이를 전송합니다.
- 각 패킷에는 데이터, 출발지/목적지의 주소, 제어 정보 등이 포함됩니다.
- 패킷은 기본적으로 큰 데이터를 전송할 때, 전체 데이터를 한번에 보내기 어렵기 때문에 사용됩니다.

## 7. WebSocket은 무엇인가?

- 실시간 양방향 데이터 통신을 위한 프로토콜입니다.
- 기존 HTTP 프로토콜은 클라이언트가 서버에 요청 보내고, 서버가 그에 대한 응답을 보내는 단방향 통신만 가능했습니다. 그러나 Websocket은 <br />
양방향 통신을 가능하게 합니다.
- 채팅, 게임, 주식 등 실시간 데이터 전송에 이용됩니다.

<br />

### 7-1. websocket은 어떻게 양방향 통신을 가능하게 하는가?

- 클라이언트가 서버에 연결 요청을 보내고, 이를 수락한 서버는 클라이언트와 소켓 연결을 맺습니다.
- 이후 연결이 끊어지지 않은 상태에서 데이터 통신이 이어지게 됩니다.

<br />

### 7-2. 3-way handshking은 무엇인가?

- 클라이언트와 서버 간에 통신을 시작하기 전에 안정적인 연결이 설정되었는지 확인하는 데에 사용 됩니다.
- 1. 클라이언트가 서버에 SYN 패킷을 보냅니다.
- 2. 서버는 클라이언트의 SYN 패킷을 받고, 이에 대한 응답으로 SYN-ACK 패킷을 보냅니다. 
- 3. 클라이언트는 SYN,ACK 패킷을 받고 ACK 패킷을 보냅니다.
- SYN은 요청의 시작, ACK는 응답이라고 볼 수 있습니다.

### 7-3. 4-way handshaking은 무엇인가?

- TCP 연결을 종료하기 위해 사용됩니다.
- 클라이언트와 서버 모두 FIN 패킷을 보내고, 각각 ACK 패킷을 받은 후에 연결이 종료 됩니다.
- 1. 클라이언트가 서버에게 연결을 종료한다는 FIN 패킷을 보냅니다.
- 2. 서버는 FIN 패킷을 받고 클라이언트가 보낸 데이터를 모두 받았다는 것을 의미하는 ACK 패킷을 보냅니다.
- 3. 서버는 더이상 보낼 데이터가 없다면 클라이언트에 FIN 패킷을 보냅니다.
- 4. 클라이언트는 FIN 패킷을 받고 이에 대해 ACK 패킷으로 응답합니다.

## 8. 쿠키와 세션의 차이점은 무엇인가?

- 쿠키와 세션은 웹 애플리케이션에서 사용자의 상태 정보를 저장하고 유지하기 위해 사용하는 방법입니다.
- 쿠키 : 클라이언트 측에 저장되는 작은 데이터 파일입니다. 서버에서 클라이언트로 쿠키를 보내면, 클라이언트는 이를 저장하고 <br />
해당 웹 사이트에 방문할 때마다 쿠키를 서버로 다시 보내게 됩니다. 쿠키는 보통 세션 ID 같은 정보를 저장하여 세션을 식별하는 데에 사용됩니다.

- 세션 : 클라이언트와 서버 간의 대화 상태를 유지하기 위한 메커니즘입니다. 세션은 클라이언트가 서버에 요청을 보내면, 서버에서 새로운 세션 ID를 <br />
생성하고, 이를 클라이언트에게 보내게 됩니다. 이후 클라이언트가 다시 요청을 보낼 때, 세션 ID를 함께 전송하여 서버에서 이전에 저장된 클라이언트 상태 <br />
정보를 가져올 수 있습니다.

- 쿠키는 클라이언트 측에 저장되고, 세션은 서버 측에서 저장됩니다. 쿠키는 클라이언트의 로컬 저장소에 저장되기 때문에 보안성이 낮아질 수 있고, 만료 기간이 <br />
존재 합니다.

<br />

### 8-1. 로컬 저장소(local storage)란 무엇인가?

- 브라우저의 저장 공간 중 하나로, 클라이언트 측에서 웹 애플리케이션의 데이터를 저장하는 데 사용됩니다.
- 로컬에 저장하므로 네트워크 지연으로 인한 성능 저하를 최소화 할 수 있습니다. 
- 사용자 설정, 인증 토큰 등을 저장할 때 활용됩니다.

### 8-2. 토큰(token)이란 무엇인가?

- 토큰은 사용자 인증을 위한 정보를 담고 있는 문자열입니다.
- 주로 서버에서 발급되며, 이후 인증이 필요한 요청에 토큰을 함께 보내 인증을 수행합니다.
- 토큰은 일반적으로 JSON Web Token(JWT) 형식으로 구성됩니다. header, payload, signature 세 부분으로 나눌 수 있습니다.
- header: 토큰 타입과 사용된 암호화 알고리즘 정보를 담고 있습니다.
- payload: 인증 정보와 추가 데이터를 담고 있습니다.
- signature: 토큰의 유효성 검증을 위한 서명 정보를 담고 있습니다.

<br />

### 8-3. Access Token과 Refresh Token는 무엇인가?

- Access Token: 사용자 인증을 위해 사용됩니다. 
- Refresh Token: Access Token을 갱신하는 데에 사용됩니다. Access Token이 만료 되지 않을 경우, 사용되지 않습니다.
- 토큰을 두가지로 나누어 관리하는 이유는 보안과 유효기간 관리 때문입니다. Access Token은 직접적인 접근 권한을 가지는 토큰으로 <br />
만약 탈취 당했을 시, 타인이 부적절한 접근을 할 수 있으므로 짧은 유효기간을 갖도록 설정합니다. 즉, access Token은 짧은 수명주기를 <br />
가지며, 자주 교체 됩니다. Refresh Token은 Access Token을 재발급 가능하기 때문에 주로 서버측에서 안전하게 보관합니다.

<br />

### 8-4. OAuth 인증 방식이란 무엇인가?

- OAuth는 클라이언트 애플리케이션이나 웹사이트에서 사용자의 권한을 대신 받아 외부 서비스의 API에 접근할 수 있는 인증 방식입니다.
- 클라이언트 애플리케이션(웹 사이트), 서비스 제공자(구글, 네이버와 같은 대형 서비스)로 나뉘어 볼 수 있습니다.
- 1. 웹사이트는 서비스 제공자의 OAuth 인증서버에 인증 요청을 합니다.
- 2. 서비스 제공자는 인증 요청을 받은 뒤, 로그인(구글이나 네이버) 페이지를 보여 줍니다.
- 3. 로그인이 완료되면, OAuth 서버는 웹사이트에 액세스 토큰을 발급해줍니다.
- 4. 웹사이트는 액세스 토큰을 사용해 안에 담긴 사용자 정보를 가져올 수 있습니다.

## 9. 캐시의 개념과 웹 성능에 미치는 영향은 무엇인가?

- 캐시는 이전에 요청한 데이터를 저장하고 다음 요청 시에 저장된 데이터를 사용하는 것을 말합니다.
- 웹에서는 주로 브라우저 캐시와 서버 캐시를 사용하여 웹 페이지 성능을 향상시킵니다.
- 브라우저 캐시 : 사용자가 방문한 웹페이지의 자원(HTML, CSS, JavaScript, 이미지 등)을 저장하여 다음에 같은 페이지를 방문할 때 서버에서 <br />
다시 자원을 요청하지 않고, 캐시에서 불러와서 보여줍니다. 이렇게 하면 페이지 로딩 속도가 빨라지고, 브라우저와 서버 간의 트래픽이 감소하여 <br />
네트워크 부하도 줄어듭니다. 웹 브라우저에 저장됩니다.
- 서버 캐시 : 서버에서 자주 요청되는 페이지나 자원을 미리 생성하여 캐시에 저장하고, 요청 시에 캐시에서 불러와서 응답합니다. 이를 통해 <br />
서버의 부하를 줄일 수 있고, 페이지 로딩 시간을 단축할 수 있습니다. 서버에 저장됩니다.

## 10. 사용자가 주소 창에 url을 입력했을 때 일어나는 일련의 과정을 설명해 보시오.

- 1. 사용자가 브라우저 주소창에 URL을 입력합니다.
- 2. 브라우저는 입력한 URL에서 도메인 이름을 추출합니다.
- 3. 추출한 도메인 이름으로 DNS 조회를 요청합니다.
- 4. DNS 서버는 도메인 이름을 IP 주소로 변환하여 응답합니다.
- 5. 브라우저는 IP 주소로 서버에 HTTP 요청을 보냅니다.
- 6. 서버는 요청을 받고, 그에 대한 응답을 보냅니다.
- 7. 브라우저는 서버에서 보내온 응답을 해석하여 화면에 렌더링 합니다.
